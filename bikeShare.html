Introduction

Bike sharing systems have gained popularity over the past decade. Because of their accessibility and affordability, more than 500 bike sharing programs are running around the world to provide a low-cost alternative for short-term commutes and to help alleviate traffic. As bike sharing demands increase, unavailability and uneven distribution of bikes become a critical issue for the programs, which could undermine usersâ€™ experience and waste public resources.
   This project implemented a dashboard using Javascript D3 library to provide interaction with the Bikeshare demand data, provided by Capital Bikeshare, the bikeshare service in Washington D.C. One of the most important goal of this project is to visualize the bike rental demands geographically and to understand how they vary overtime.
   The result of this project can serve as part of city mobility research and provides insights of how to make bikeshare service better and benefit more people's daily life.

BikeShare Data

This project worked on two bikeshare datasets. The first one is trip history data, hosted by Capital Bikeshare, where each row contains starting station, ending station, starting time, ending time and duration of a trip.
  The second one is the aggregated bike demand data by hour along with the extraneous variables such as temperature, wind speed, and humidity. It is hosted by UCI ML repository.
  Subject to the limited storage space provided by web service, I partitioned the data by months and only use the data in November 2012.

Data Preparation

To calculate hourly demands for each station, I first aggregated data by hours and stations to calculate how many bikes come and leave. Then, minus number of leaving with number of coming to obtain the hourly demands. In addition, to be able to plot them on the map, I merge the latitude and longitude into the data. Finally, I merged my aggregated dataset with the UCI dataset to incorporate the extraneous variables.

Techniques

Calendar and Clock:

They are filters. Click them to select different part of data.

Symbol Map

I encoded the locations and bike demands of each station in the circles and use contrasting color to distinguish the supply and demand. Green stands for the excess of supplies while red stands for the excess of demands. In addition, I encoded intensity in the opacity of colors. So, the stronger the color is, the more excess the value is.
   The interactivity of this map includes a tooltip that displays the name of a bike station and its bike demand at certain time stamp. To access more detail about certain stations, we can click the circles and the line chart would show their demand distribution over 24 hours.

Two Sided barchart

This visualization aims to provide bike demand information by region. I decide to implement it using vertical two-sided bar chart, which allows users to easily compare values of different regions with the least lie factor involved.
   For the interactivity, hover bars to see the exact bike demands and their location.

Line Chart

The purpose of this visualization is to realize the distribution of bike rental demand and also allow users to make easy comparison between two selected stops.
   For the interactivity, I implemented tooltip cursors for each time series. To display it, hover the mouse in plotting region.

Challenge

The biggest challenge of this project is to perform simultaneous transition of symbol map, bar chart, and line chart when the calendar or the clock is clicked. I addressed this challenge with function objects and sets of associated object methods. For instance, for each of them, I define one function object to initiate the plot, one object method to update the underlying data attached with svg elements and another object method to manipulate the transition. Once each function object is instantiated, I can easily handle the transition by simply calling their object methods.
   In addition, number of station with observed records varies from day to day. For Nov. 1st, there might be 95 stations while for Nov. 2nd, there might be only 90 stations. To dynamically track which stations have records on a given date, I assigned a key function of station name in data(). It would authomatically subset my dataset into enter, exit, and update. With separated accesses of this three subsets, I can create new circles for enter data, update the color scale and opacity for update data, and remove the circles of exit data.

Conclusion

Generally, we can classify each station into either a more downtown neighborhood such as zip code 20004, 20005, 20006, 20007, 20245, 20415, 20433, and 20546 and a more residential area like 20001, 20002, 20003, 20008, 20009, 20010, 20011. Stations within the residential regions have huge demand during 7am to 9am in weekdays, the time when people leave for works or schools. This demand not surprisingly delays to 10am to 12pm in weekends.
   Conversely, stations within the downtown area suffer a huge bike in-flood every morning. This uneven distribution would reverse in the afternoon when people start to leave their school or office.
   Consequence of this uneven distribution problem might be lack of decks when people try to return bikes or might be lack of bikes when people actually need ones. Either way could be very frustrating and ruins user experience. Hence, I think with the solid understanding of bike demand distribution of each station the bikeshare service can develop a redistribution program to move those bikes around in rush hours and alleviate this problem.
